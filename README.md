## Что такое Git?

**Git** - это распределённая система контроля версий, созданная Линусом Торвальдсом в 2005 году. Она позволяет отслеживать изменения в файлах, координировать работу нескольких разработчиков над одним проектом и при необходимости откатываться к любой предыдущей версии кода.

В отличие от централизованных систем, где история хранится только на одном сервере, Git является **распределённой** системой: каждый разработчик имеет полную копию репозитория со всей историей изменений. Благодаря этому работа с кодом возможна даже без подключения к интернету, а риск потери данных существенно снижается.

## Ключевые понятия

| Понятие | Описание |
|---|---|
| **Репозиторий** (repository, repo) | Место хранения всех файлов проекта вместе с полной историей изменений. Бывает локальным (на компьютере) и удалённым (на сервере — GitHub, GitLab и т.д.) |
| **Коммит** (commit) | Снимок состояния проекта в конкретный момент времени. Содержит уникальный хэш, данные автора, дату и сообщение с описанием изменений |
| **Ветка** (branch) | Независимая линия разработки внутри репозитория. По умолчанию — `main` или `master` |
| **Рабочая директория** (Working Directory) | Папка на компьютере, где находятся файлы проекта |
| **Индекс / Область подготовки** (Staging Area) | Промежуточная область, куда помещаются изменения перед созданием коммита |
| **HEAD** | Указатель на текущий коммит или ветку |
| **Тег (Tag)** | Метки, используемые для фиксации определенных точек в истории (обычно релизов, например, v1.0). В отличие от веток, они не изменяются после создания |

## Команды

### Инициализация и клонирование

**`git init`** - создаёт новый пустой репозиторий в текущей папке (появляется скрытая папка `.git`):

```bash
git init
git init my-project    # создать репозиторий в новой папке my-project
```

**`git clone`** - копирует удалённый репозиторий на локальный компьютер вместе со всей историей:

```bash
git clone https://github.com/username/repository.git
git clone https://github.com/username/repository.git my-folder
```

---

### Ежедневная работа

**`git status`** - показывает текущее состояние рабочей директории и области подготовки: какие файлы изменены, добавлены в индекс или не отслеживаются.

**`git add`** - перемещает изменения из рабочей директории в область подготовки:

```bash
git add filename.txt   # добавить конкретный файл
git add .              # добавить все изменения в текущей папке
git add src/           # добавить все файлы в папке src
git add *.js           # добавить все .js файлы
```

**`git commit`** - сохраняет подготовленные изменения в историю репозитория:

```bash
git commit -m "Добавить валидацию формы входа"
```

**`git log`** - выводит список всех коммитов в текущей ветке от новейшего к старейшему (хэш, автор, дата, сообщение).

**`git diff`** - показывает разницу между версиями файлов (`+` — добавленные строки, `-` — удалённые):

```bash
git diff                    # изменения не в индексе
git diff --staged           # изменения, добавленные в индекс
git diff HEAD~1             # сравнить с предыдущим коммитом
git diff main feature-branch  # сравнить две ветки
```

### Работа с ветками

**`git branch`** - управление ветками:

```bash
git branch -a               # все ветки, включая удалённые
git branch feature-branch   # создать ветку feature-branch
git branch -d feature-branch  # удалить слитую ветку
git branch -D feature-branch  # принудительно удалить ветку
```

**`git switch`** - переключение между ветками (рекомендуется с Git 2.23+):

```bash
git switch main             # переключиться на ветку main
git switch feature-branch   # переключиться на ветку feature-branch
```

**`git merge`** - объединяет изменения из одной ветки в другую:

```bash
git switch main             # переключиться на основную ветку
git merge feature-branch    # слить feature-branch в main
```

**`git rebase`** - переносит коммиты из одной ветки поверх другой, создавая более линейную историю:

```bash
git switch feature-branch   # переключиться на ветку feature-branch
git rebase main             # перебазировать на main
```

### Удалённые репозитории

**`git remote`** - управление ссылками на удалённые репозитории:

```bash
git remote -v                                 # посмотреть все удалённые репозитории
git remote add origin https://...             # добавить удалённый репозиторий
git remote remove origin                      # удалить ссылку
git remote rename origin upstream             # переименовать
```

**`git fetch`** - загружает изменения с удалённого репозитория, но не применяет их к локальным файлам:

```bash
git fetch origin
git fetch --all    # получить изменения со всех удалённых репозиториев
```

**`git pull`** - загружает изменения и сразу применяет их (`fetch` + `merge`):

```bash
git pull origin main       # получить и применить изменения из ветки main
git pull --rebase          # получить изменения с перебазированием
```

**`git push`** - отправляет локальные коммиты на удалённый репозиторий:

```bash
git push origin main       # отправить ветку main на сервер
```

### Отмена изменений

**`git restore`** - отменяет изменения в рабочей директории или убирает файлы из индекса:

```bash
git restore filename.txt   # отменить изменения в файле
git restore .              # отменить все изменения в рабочей директории
```

**`git reset`** - перемещает указатель HEAD на другой коммит. Три режима:

```bash
git reset --soft HEAD~1    # отменить коммит, сохранить изменения в индексе
git reset --mixed HEAD~1   # отменить коммит, убрать из индекса
git reset --hard HEAD~1    # полностью удалить коммит и все изменения
```

**`git revert`** - создаёт новый коммит, отменяющий изменения указанного. Безопасен для общих веток, так как не изменяет историю:

```bash
git revert 123abc          # создать коммит, отменяющий коммит 123abc
git revert HEAD            # отменить последний коммит
```

---

### Временное сохранение изменений (Stash)

**`git stash`** позволяет «отложить» незакоммиченные изменения, чтобы переключиться на другую задачу:

```bash
git stash                      # сохранить изменения в stash
git stash list                 # посмотреть список stash
git stash pop                  # применить последний stash и удалить его
git stash apply stash@{1}      # применить конкретный stash без удаления
git stash drop stash@{0}       # удалить конкретный stash
git stash clear                # удалить все stash
```

### Теги

**`git tag`** - маркировка важных точек в истории проекта (например, релизов):

```bash
git tag                            # показать все теги
git tag v1.0.0                     # создать лёгкий тег
git tag -a v1.0.0 -m "Релиз 1.0.0"  # создать аннотированный тег
git tag -a v1.0.0 abc1234          # тег на конкретный коммит
```

По умолчанию `git push` не отправляет теги на сервер - это нужно делать явно:

```bash
git push origin v1.0.0            # отправить один тег
git push origin --tags            # отправить все теги
git tag -d v1.0.0                 # удалить тег локально
git push origin --delete v1.0.0   # удалить тег на сервере
```

## .gitignore

Далеко не все файлы должны попадать в репозиторий. Зависимости, временные файлы, логи, файлы с паролями и ключами API - всё это не должно храниться в истории Git.

Для исключения ненужных файлов в корне проекта создаётся файл `.gitignore`. Каждая строка - это шаблон файлов или папок, которые Git будет игнорировать. Файл `.gitignore` сам должен быть добавлен в репозиторий, чтобы все участники команды работали с одинаковым набором исключений.

> **Важно:** создавать `.gitignore` необходимо в самом начале проекта. Git не игнорирует файлы, которые уже были добавлены в репозиторий ранее.

Пример `.gitignore`:

```
# Зависимости
node_modules/
venv/

# Логи
*.log

# Файлы окружения
.env
.env.local

# Сборки
/dist
/build
```
